# Hermes Agent 安全功能分析（代码级 + 产品级安全全景）

> 本文逐一分析 Hermes Agent 的安全相关功能，**每项机制均对应真实文件 file:line**（已通过源码核对）。配套图：`security-panorama.svg`（浏览器打开）。
>
> **核心前提（来自 `SECURITY.md` §2.2）**：Hermes 明确声明 **唯一真正的安全边界是操作系统**。进程内的审批门、输出脱敏、模式扫描、工具白名单 **都是启发式（heuristics），不是边界（boundaries）**。理解这点是看懂整张安全全景的钥匙——进程内机制防"协作模式下的事故"，OS 级隔离防"对抗性 LLM"。

## 0. 安全威胁模型总览

| 威胁 | 攻击面 | Hermes 对应控制 | 是否边界 |
|------|--------|----------------|----------|
| 危险 Shell 命令（rm -rf / 关机 / fork bomb） | terminal 工具 | Hardline 阻断 + 危险模式检测 + Tirith | 启发式 |
| LLM 越权执行代码逃逸到宿主 | terminal / code-exec | 6 种执行后端隔离（docker/ssh/...） | **OS 边界** |
| 未授权用户通过 IM 平台操控 Agent | gateway | authz_mixin 允许列表 + pairing 配对 | **边界（外部面）** |
| 密钥泄漏进日志 / prompt / 工具输出 | 全局 | redact.py 正则脱敏 | 启发式 |
| 跨 profile 密钥串号（多租户网关） | gateway 多路复用 | secret_scope fail-closed 隔离 | **边界（进程内租户）** |
| 借入的外部密钥落盘 | auth.json | credential_persistence 指纹化清洗 | 边界（落盘面） |
| 路径穿越 / 越界写 | 文件工具 | path_security + file_safety 黑名单 | 启发式 |
| SSRF / 访问云元数据 | web / browser | url_safety 私网+元数据阻断 | 启发式（含强制底线） |
| 供应链恶意依赖（PyPI/npm） | MCP 启动 / 安装 | osv_check（OSV MAL-*）+ security_audit | 启发式 |
| 恶意技能代码 | skills 安装 | skills_guard（98 模式）+ AST 审计 | 启发式（边界=人工审查） |
| 恶意 MCP 服务器配置 | MCP 配置 | mcp_security IOC + 持久化形态检测 | 启发式 |
| 提示注入（context/memory） | 上下文文件 | threat_patterns（28 模式 + 隐形 Unicode） | 启发式 |
| TLS 降级 / CA 损坏 | 模型调用 | ssl_guard 启动校验 | 防御加固 |
| 配额耗尽 / 429 重试放大 | 模型调用 | nous_rate_guard 跨会话熔断 | 成本保护 |

## 1. 命令执行安全（Command Execution Gate）

入口：`tools/terminal_tool.py:2088` → `_check_all_guards()` → `tools/approval.py:check_all_command_guards()` (`approval.py:1470`)。这是 terminal 工具执行前的**唯一总闸**。

### 1.1 Hardline 无条件阻断（最底线）
- **文件**：`tools/approval.py:333` `detect_hardline_command()`；模式 `HARDLINE_PATTERNS_COMPILED` (`:293`)，12 条（`:262-284`）。
- **覆盖**：`rm -rf /`、`/home /etc /usr`、`rm -rf ~`、`mkfs*`、`dd of=/dev/sd*`、fork bomb `:(){...}`、`shutdown/reboot/halt/poweroff`、`systemctl poweroff`、`init 0/6`。
- **特性**：**连 `--yolo`、`approvals.mode=off`、cron approve 都无法绕过**（`:221-244`）。仅对可伤害宿主的后端生效（local/ssh/容器宿主 cron）；容器化后端（docker/singularity/modal/daytona）直接跳过整个检查（`check_all_command_guards` 开头 `:1470`）。

### 1.2 sudo stdin 密码爆破阻断
- **文件**：`tools/approval.py:314` `_check_sudo_stdin_guard()`；正则 `_SUDO_STDIN_RE` (`:309`)。
- **逻辑**：检测 `sudo -S`（从 stdin 喂密码）且未配置 `SUDO_PASSWORD` 环境变量 → 无条件阻断（在 YOLO 之前，`:1497`）。

### 1.3 危险模式检测（可审批）
- **文件**：`tools/approval.py:657` `detect_dangerous_command()`；`DANGEROUS_PATTERNS` 47 条（`:380-522`）：文件变更、DB 破坏（DROP/DELETE 无 WHERE/TRUNCATE）、系统配置编辑、进程终止、RCE（`curl|sh`）、提权、git 破坏性操作、SSH/凭证文件。
- **反混淆归一化**：`_normalize_command_for_detection()` (`:559`) — 去 ANSI、去空字节、NFKC（全角→ASCII）、去反斜杠转义、去空串字面量 `r''m→rm`、家目录/HERMES_HOME 路径回写为 `~`。
- **审批范围**：once / session（`approve_session` `:770`）/ always（`approve_permanent` `:835` → 落 `config.yaml` `command_allowlist`）。

### 1.4 Tirith 内容级扫描（外部 Rust 二进制）
- **文件**：`tools/tirith_security.py:697` `check_command_security()`；自动安装 `ensure_installed()` (`:599`)。
- **作用**：AST 级命令分析，捕捉正则漏掉的 homograph URL、shell 注入、管道喂解释器、终端注入。退出码 `0=allow/1=block/2=warn`。
- **配置**：`security.tirith_enabled`（默认 True）、`tirith_timeout`（默认 5s）、`tirith_fail_open`（默认 **True** — 超时/出错放行，仍有模式门兜底）。下载校验 SHA-256 + 可选 cosign 签名。
- **特例**：`.app` TLD 误报抑制 `_is_app_tld_finding()` (`:808`)。

### 1.5 Smart 审批（辅助 LLM）
- **文件**：`tools/approval.py:1136` `_smart_approve()`（`approvals.mode=smart` 时）。
- **抗注入**：剥离 shell 注释（防 `# APPROVE` 注入，`:1159`）、XML 定界包裹（`:1181`）、`temperature=0 max_tokens=16`。输出 APPROVE/DENY/escalate。

### 1.6 网关审批排队（并发会话）
- **文件**：`tools/approval.py:690` `_ApprovalEntry`、`_await_gateway_decision()` (`:1352`)、`resolve_gateway_approval()` (`:729`)。
- **逻辑**：阻塞 Agent 线程等待用户 `/approve`/`/deny`；超时 `approvals.gateway_timeout`（默认 300s）；每秒轮询并发心跳防 watchdog 超时。

## 2. 执行隔离 / 沙箱（唯一真正的 OS 边界）

抽象基类 `tools/environments/base.py:287` `BaseEnvironment`，6 种后端：

| 后端 | 文件 | 隔离强度 | 关键加固 |
|------|------|----------|----------|
| **Local** | `environments/local.py:572` | **无**（直接跑宿主） | 仅剥离 Hermes 密钥环境变量 `_sanitize_subprocess_env()` (`:208`)；进程组 `os.setsid()` 超时 `killpg` |
| **Docker** | `environments/docker.py:503` | 容器 | `--cap-drop ALL` + 精选 `--cap-add`、`--security-opt no-new-privileges`、`--pids-limit 256`、tmpfs `nosuid`、`--network=none`（可选）、孤儿容器回收 `:138` |
| **SSH** | `environments/ssh.py:36` | 远程主机 | ControlMaster 复用；文件同步防符号链接逃逸 `relpath` 校验 `:225-234` |
| **Singularity** | `environments/singularity.py:158` | 容器 | `--containall --no-home`、overlay/tmpfs |
| **Modal** | `environments/modal.py:164` | 云 Serverless 沙箱 | 网络+宿主完全隔离；快照持久化 |
| **Daytona** | `environments/daytona.py:30` | 云 Serverless 沙箱 | 同上；自动 resume |

**Local 后端密钥剥离细节**：`_HERMES_PROVIDER_ENV_BLOCKLIST`（`local.py:96-191`）剥离 LLM API key、IM token、Bedrock 凭证；但 **AWS 凭证刻意保留**（`:85-95`，因用户自己的 shell 本就有同等访问，见 SECURITY.md §3.2）；可经 `tools.env_passthrough` 重新放行。

**SECURITY.md 定义的两种受支持隔离姿态**：
1. **终端后端隔离**（§2.2）：非默认后端把 LLM 发出的 shell/文件工具关进容器/远程/云沙箱。**不约束**：Python 进程本身、code-exec 工具、MCP 子进程、插件加载、hook、技能加载。
2. **整进程包裹**（§2.2）：整个 Agent 进程树进沙箱（Docker Compose 或 NVIDIA OpenShell 声明式策略）。**当 Agent 摄入不可信内容（开放网络/入站邮件/多用户频道/不可信 MCP）时，这是受支持姿态**。

## 3. 网关授权（多租户外部面 — 边界）

入口：`gateway/run.py` 收到入站消息 → `gateway/authz_mixin.py:176` `_is_user_authorized()`。

### 3.1 多层授权（按序，`authz_mixin.py`）
1. 系统事件豁免（HomeAssistant/webhook，已在适配器认证）`:193`
2. 群组白名单 `TELEGRAM_GROUP_ALLOWED_CHATS` 等 `:208`
3. 平台级 `{PLATFORM}_ALLOW_ALL_USERS` `:293`
4. Discord 角色授权 `role_authorized is True` `:298`
5. Bot 白名单 `DISCORD_ALLOW_BOTS` `:306`
6. **配对存储 `pairing_store.is_approved()`（无论是否配白名单都查）** `:311`
7. 平台用户白名单 `{PLATFORM}_ALLOWED_USERS` / `GATEWAY_ALLOWED_USERS` `:316`
8. WhatsApp 手机↔LID 别名解析 `:425`
9. **默认拒绝（fail-safe deny）**

### 3.2 配对系统（动态授权）
- **文件**：`gateway/pairing.py`。`generate_code()` (`:204`) 8 位 `secrets.choice`；SHA-256+盐存储（**明文不落盘**）；1h TTL；限流 10min/次；5 次失败锁定 1h；`secrets.compare_digest()` 常量时间比对（`:307`）；文件 `chmod 0600`。

### 3.3 斜杠命令访问控制
- **文件**：`gateway/slash_access.py`。两层：能否对话 + 能跑哪些命令。`allow_admin_from`（管理员）、`user_allowed_commands`（非管理员）、永久底线 `{help, whoami}` (`:50`)。仅当显式配 `allow_admin_from` 才启用（`:188`）。

### 3.4 跨平台身份
- WhatsApp：`gateway/whatsapp_identity.py` JID 归一化 `:48`、别名图遍历 `:121`、`_SAFE_IDENTIFIER_RE` 防路径穿越 `:43,152`。

**SECURITY.md 外部面 5 条规则（§2.6）**：①每个跨信任边界的面都需授权；②每个网络暴露适配器都需白名单，**fail-closed**（无白名单=拒绝派发）；③Session ID 是路由句柄非授权边界；④授权集合内所有调用者等同信任（要能力隔离请用独立 Agent 实例）；⑤绑非 loopback（`--host 0.0.0.0`）是 break-glass 操作员决定。

## 4. 密钥与凭证安全

### 4.1 输出脱敏 redact.py
- **文件**：`agent/redact.py`。48 条厂商前缀模式 `_PREFIX_RE` (`:208`)：`sk-`、`ghp_`、`github_pat_`、`xox[baprs]-`、`AIza`、`xai-`、Stripe/AWS/HF/npm/PyPI。另：敏感参数（access_token/password/...）、Authorization 头（保留 scheme）、JWT `eyJ...`、DB 连接串密码、私钥块。
- **掩码策略**：`mask_secret()` (`:213`) — >18 字符保留首尾 4 位，短则全掩 `***`。
- **强制点**：日志格式化器 `RedactingFormatter`（`hermes_logging.py:513`，applied at agent/errors/gateway 各 handler）；发 LLM 前消息内容（`run_agent.py:1389-1418`，含 system prompt）；工具输出 JSON（`agent/agent_runtime_helpers.py` 用 `force=True`）。
- **开关**：`HERMES_REDACT_SECRETS`（默认 **on**）/ `security.redact_secrets`。

### 4.2 密钥作用域隔离 secret_scope.py（多租户 fail-closed）
- **文件**：`agent/secret_scope.py`。`ContextVar` 每任务密钥映射 (`:55`)；`get_secret()` (`:123`) 解析逻辑：全局变量直读 os.environ → 有 scope 读 scope（**缺失不回退**）→ **无 scope 且 multiplex 激活则抛 `UnscopedSecretError`（fail-closed）** → 否则回退 os.environ（单 profile 兼容）。
- **强制点**：网关每轮 `_profile_runtime_scope()`（`gateway/run.py:1395` `set_secret_scope(build_profile_secret_scope(...))`）；`gateway.multiplex_profiles=true` 时调 `set_multiplex_active(True)`。

### 4.3 借入凭证落盘清洗 credential_persistence.py
- **文件**：`agent/credential_persistence.py`。白名单 `_PERSISTABLE_SOURCES` (`:20`) 仅 Hermes 自有 OAuth（anthropic/hermes_pkce、minimax-oauth、nous/device_code、openai-codex、xai-oauth）可落原始密钥；其余"借入源" → `sanitize_borrowed_credential_payload()` (`:151`) 计算 SHA-256 指纹 `sha256:{16hex}` 并剥离所有 secret 值键。
- **强制点**：`credential_pool.py:201` `to_dict()` 落盘前清洗。

### 4.4 OAuth 令牌管理（防单用 token 复用/竞态）
- **文件**：`tools/mcp_oauth.py`（`HermesTokenStorage`，token 落 `~/.hermes/mcp_servers/<n>/tokens.json` 权限 0o600）、`tools/mcp_oauth_manager.py`（磁盘 mtime 监测外部刷新 `:74`、401 去重 `pending_401` `:76`）。凭证池刷新 `credential_pool.py:861` `_refresh_entry()` 单用 refresh token 防护：刷新前先从 auth.json 同步检测他进程是否已消费；终态错误（`token_revoked/invalid_grant/refresh_token_reused`）转 `STATUS_DEAD` (`:69`) 永久排除。多进程 `_auth_store_lock()` 串行化。

## 5. 文件系统与网络安全

### 5.1 路径穿越 path_security.py
- **文件**：`tools/path_security.py`。`validate_within_dir(path, root)` (`:15`) — 双向 `resolve()`（跟随符号链接+归一化 `..`）后 `relative_to()` 校验；`has_traversal_component()` (`:37`) 快查 `..`。
- **强制点**：cronjob_tools:431、skills_tool:1210、credential_files:87/156、skill_manager_tool:517、file_tools:1363。

### 5.2 写/读黑名单 file_safety.py

- **强制点**：web_tools:967（`async_is_safe_url`）、browser_tool:2469/2479/2544。

### 5.4 网站黑名单 website_policy.py

## 6. 供应链与扩展信任

### 6.1 依赖恶意检测 osv_check.py
- **文件**：`tools/osv_check.py`。`check_package_for_malware()` (`:26`) — 从命令推断生态（npx→npm，uvx/pipx→PyPI），解析包名+版本，POST `api.osv.dev/v1/query`，**仅筛 `MAL-*` 恶意公告**（非全部 CVE）。10s 超时 **fail-open**。
- **强制点**：MCP stdio 启动前 `mcp_tool.py:1776`（每次连接重查），命中 `raise ValueError` 阻止 spawn。

### 6.2 供应链漏洞审计 security_audit.py（按需）
- **文件**：`hermes_cli/security_audit.py`。扫 venv（importlib.metadata 全量）+ 插件（requirements/pyproject 精确 pin）+ MCP（npx/uvx 版本）→ OSV `querybatch`。CLI `hermes security audit`，`--fail-on {low..critical}` 命中则 exit 1。8 线程并行取详情。

### 6.3 技能静态扫描 skills_guard.py
- **文件**：`tools/skills_guard.py`。`THREAT_PATTERNS` **98 条**跨 17 类（`:96-516`）：exfiltration/injection/destructive/persistence/network/obfuscation/privilege_escalation/credential_exposure 等。`scan_skill()` (`:627`) 递归扫文本文件 + 隐形 Unicode 检测 + 结构检查（文件数/大小/符号链接逃逸）。
- **裁决** `_determine_verdict()` (`:1064`)：critical→dangerous、high→caution、否则 safe。
- **信任分级安装策略** `should_allow_install()` (`:686`)：builtin（全允）/ trusted（openai/anthropic/HF/NVIDIA 签名，safe→caution 允、dangerous 阻）/ community（safe 允、caution+ 阻）/ agent-created（dangerous→ask）。`--force` **不能**越过 community/trusted 的 dangerous。
- **完整性**：SHA-256 内容哈希 `content_hash()` (`:766`)。强制点 `skills_hub.py:495` 安装、`web_server.py` 仪表盘、`:1035` 更新、`:1409` 自审。

### 6.4 技能 AST 深度审计 skills_ast_audit.py（仅诊断）
- **文件**：`tools/skills_ast_audit.py`。AST visitor (`:33`) 标记 dynamic_import / `__import__` 计算式 / `getattr` 非字面量 / `__dict__` 访问。`hermes skills audit --deep` 触发。**不阻断**，仅提示。

### 6.5 MCP 服务器信任 mcp_security.py
- **文件**：`hermes_cli/mcp_security.py`。`validate_mcp_server_entry()` (`:121`)：① IOC 硬黑名单 `_IOC_SUBSTRINGS`（攻击者 SSH 公钥片段、"hermes-0day"、攻击源 IP）`:80`；② 仅当 command 是 shell 解释器才查模式：网络外联 `_EGRESS_PATTERN`（curl/wget/nc/socat//dev/tcp/）+ 外泄提示，OS 持久化 `_PERSISTENCE_PATTERN`（authorized_keys/.ssh//etc/sudoers/cron/.bashrc）。
- **强制点（双重）**：配置保存时（dashboard/CLI/gateway 写入前校验，抛异常拒写）+ 加载时 `mcp_tool.py:3084` `_filter_suspicious_mcp_servers()`（日志告警+跳过该服务器）。

### 6.6 工具 Schema 净化 schema_sanitizer.py（防后端崩溃/注入）

## 9. 漏洞披露与范围（SECURITY.md §3, §5）

**In Scope（视为漏洞）**：①逃离声明的 OS 隔离姿态；②未授权外部面访问；③应阻止却发生的凭证外泄；④信任模型文档违背。

**Out of Scope（不视为漏洞）**：①绕过进程内启发式（审批门/脱敏/Skills Guard）；②单纯提示注入（未链到 §3.1 后果）；③所选隔离姿态的后果（local 后端下 shell 触达宿主）；④文档化的 break-glass（`--insecure`）；⑤社区技能/插件（=操作员审查面）；⑥无外部控制的公网暴露；⑦允许 shell 的姿态下的工具级读写限制。

**披露**：GitHub Security Advisories 或 security@nousresearch.com；不公开 issue；无 bug bounty；协调披露 90 天。

## 10. 安全设计哲学总结

1. **单一边界原则**：进程内一切是启发式，OS 隔离是唯一边界。诚实标注降低误用。
2. **纵深防御**：危险命令 = Hardline(不可绕) → sudo guard(不可绕) → Tirith → 模式 → 审批，五层。
3. **Fail-closed 关键面**：网关无白名单拒绝派发；多租户 secret_scope 无 scope 抛异常；SSRF DNS 失败拒绝。
4. **Fail-open 容错面**：Tirith/OSV/website_policy 出错放行（有其他层兜底，避免可用性崩塌）。
5. **信任分级**：技能/MCP 按来源（builtin/trusted/community）施加不同安装策略。
6. **人工审查是技能/插件的真边界**：代码明确导向"装前读 Python"。
7. **成本即安全**：rate guard 把配额耗尽/重试放大当作要防御的滥用。
